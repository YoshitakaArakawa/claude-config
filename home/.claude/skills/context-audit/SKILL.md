# Context Audit

## 目的

Agent のコンテキストに入る資産が「リポの目的に対して過不足なく・矛盾なく・正しい層に」配置されている状態を保つ。この Skill はその状態からの乖離を検出し、人間が読める報告にまとめる。細かい手順は規定しない。以下のゴールと観点に照らし、調査の進め方は対象リポの構成に合わせて判断する。

## 進め方

1. **監査対象を特定する**：固定リストにせず、コンテキストに入る経路から洗い出す——常時ロード（CLAUDE.md・AGENTS.md 等）、オンデマンド（skills 配下一式）、イベント駆動（hooks とその実行スクリプト）、subagent 定義（agents 配下）。リポ固有の配置（ミラー構成・wrapper 等）は README / CLAUDE.md から把握する
2. **観点に沿って監査する**（次節）
3. **報告する**（次々節）。修正はこの後
4. **ユーザーが選んだ項目だけ修正し、検証する**：旧記述の残存 grep が 0 件になることを確認する。Skill を編集した場合は creating-skills 同梱の validate-skill.py が 0 errors になることも確認する

## 監査の観点

重要度順。発見の分類にもこの並びを使う。

- **バグ（実害あり）**：記述どおりに動かないもの。例：frontmatter のパース失敗（description 中のコロン + スペース等）で description が空になる、hooks が参照するスクリプトの実体不在、リンク切れの参照
- **stale**：実体に追従していない記述。例：文書中の個数・番号・ファイル名・構成説明が、改変後も古いまま残っている
- **矛盾**：資産間で規範が食い違うもの。例：同種の Skill 群で一部にだけガード文がある、文書上の配置規則と実際の配置の乖離
- **重複**：同じ知識が複数箇所に書かれ drift リスクを生むもの。対応案は「正典を 1 箇所に決め、他を参照化する」を基本にする
- **層の誤配置**：知識が適切な層にないもの。常時適用すべき規範 → CLAUDE.md、オンデマンドで呼ぶ知識・手順 → Skill、イベント駆動で強制する動作 → hooks
- **Agent に不要な記述**：人間向けの備忘・開発経緯・過去の物語がコンテキストを消費しているもの

Skill ファイル単体の品質（description の書き方・冗長性・可搬性）の判定基準は creating-skills の references/writing-rules.md を正典とし、ここには再掲しない。

## 報告

- 修正前に必ず報告し、ユーザーの選択を待つ
- 人間が読む文章で書く：結論を先頭に置き、観点ごとに節を分け、各発見は「何が・どこで・なぜ問題か・対応案」が一読で分かる散文にする。パスの羅列やログ転記で済ませない
- 問題が見つからなかった観点は「問題なし」と明記する。指摘を作るために些末な指摘を膨らませない
- この場で検証できない事項（fresh session でしか確認できないトリガ挙動等）は「未検証」と明示する
