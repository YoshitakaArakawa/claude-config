# Readability Audit

## 目的

読者の認知負荷を下げ、注意の配分を設計する——この目的を Markdown 文書に適用する。Agent 向け資産も人間が確認・レビューする文書であり、対象に含む。診断だけで終わらせず、読みやすい形への編集までを担う。判定基準は本文に集約し、細かい手順は対象文書に合わせて判断する。

## 進め方

1. **対象と読者を確認する**：対象ファイルは引数または会話から特定する。読者（初見の利用者か、繰り返し参照する運用者か）と文書の種類（参照文書か・解説文書か・Agent 向け資産か）で基準の重みが変わる（判定基準の末尾）
2. **判定基準に照らして診断する**
3. **編集し、変更点と理由を報告する**：git 管理下の文書は、診断結果の要約を添えてそのまま編集してよい。git 管理外、または構成の組み替えを伴う大規模な書き換えは、編集前に方針を提示して承認 1 問を挟む

## 判定基準

**構造——読者は全文を読まない前提で組む**

- 結論・文書の目的を冒頭に置く。リード文は 1〜2 文に絞る
- 長い文書には目次を置き、見出しだけ拾い読みして筋が通る状態にする
- 各節も結論先頭。調査ログ・経緯・長い補足は details に折り畳む

**文と段落——密度に波を作る**

- 一文一義。読点で主張をつなぎ始めたら文を切る。短くするのは分割であって削除ではなく、文脈は削らない
- 密な段落 2〜3 個ごとに、確定事項を固定する疎な一行を置く
- 具体（コード・数値・ファイル名）と意味づけ（それが何を意味するか）を交互に置く
- 並列 3 項目以上は箇条書きにする。順序があるなら番号リスト、属性の比較なら表。重要度の違う項目を同じテンプレートで並べない

**文書実況の排除**

- 各文が更新しているのは「内容」か「文書の進行」かを判定する。「この章では〜を説明する」「次に〜を見る」のような進行実況は削除対象。節の役割は見出しと内容自身に語らせる

**重複と正典**

- 詳述が別文書（正典）にあるなら、要約＋リンクに置き換えて一本化する。同じ知識の二重記載は drift の温床

**編集時の制約**

- 外部から参照されている見出しアンカーを壊さない。壊す場合は参照元も追従させる
- 用語・自称（「本リポジトリ」等）を文書内で統一する
- Agent 向け資産を編集した場合は creating-skills 同梱の validate-skill.py を実行し、0 errors を確認する。description を変更した場合はトリガー回帰が必要になるため、その場で確認できなければ「未検証」として報告に明示する

**文書の種類による重み**

- README・リファレンス：スキャン耐性と結論先頭を最優先する
- ブログ・解説記事：問いを先に立てて後で回収する緊張の設計や、断定と逡巡の交互配置といったリズムの技法も使ってよい
- Agent 向け資産（SKILL.md・references・CLAUDE.md）：本文の毎行が恒常的なトークンコストになるため、行数を増やす編集（目次・リード文・疎の一行の挿入）は行わず、短くする・分割する方向に限定する。一文一義・規範が先・列挙の型・用語統一は通常どおり適用する

文書実況の排除はすべての種類に共通して適用する。

## 報告

- 変更点は「どこを・どう変えた・どの基準に基づくか」が一読で分かる散文で書く
- 直さなかった指摘（意図的に残した密集・原文尊重等）は理由付きで明示する
- 問題が少ない文書に無理に指摘を作らない。「概ね問題なし」で終えてよい

## 出典

判定基準のうち認知リズムに関する部分（密度の波・具体と意味づけの交互配置・文書実況の排除・緊張の設計）は、k16shikano 氏の cognitive-rhythm-writing（https://gist.github.com/k16shikano/eb2929f13ed19c97188393d297be8432）の趣旨を文書編集向けに書き起こしたもの。原文の転載ではない。
